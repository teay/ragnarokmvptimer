# Performance Optimization Plan: Ragnarok MVP Timer
แผนการเพิ่มประสิทธิภาพและความเร็วเพื่อให้แอปพลิเคชันทำงานได้ลื่นไหลและเบาเครื่องที่สุด

## 1. แนวคิดหลัก (Core Philosophy)
**"Measure First, Fix Later"** — เราจะไม่คาดเดาจุดที่หน่วง แต่จะใช้เครื่องมือวัดผลให้ชัดเจน เพื่อแก้ไขให้ตรงจุดโดยไม่เพิ่มความซับซ้อนให้โค้ดโดยไม่จำเป็น

---

## 2. การตรวจจับและวัดผล (Analysis & Diagnostics)
เพื่อหาจุดบกพร่องที่ทำให้แอปหน่วง (Jank) หรือใช้ CPU สูง:
- **React Profiler:** ตรวจสอบ "การเรนเดอร์ที่ซ้ำซ้อน" (Unnecessary Re-renders) หากเปลี่ยนเวลาตัวเดียวแล้ว Component อื่นขยับตาม แสดงว่ามีคอขวด
- **Chrome DevTools (Performance Tab):** บันทึกจังหวะที่รู้สึกหน่วง เพื่อดูว่ามี Script ตัวไหนประมวลผลนานเกิน 16ms (Long Tasks)
- **Lighthouse Audit:** วัดประสิทธิภาพโดยรวมตามมาตรฐานเว็บ (LCP, TBT, CLS)

---

## 3. ยุทธศาสตร์การแก้ไข (Technical Strategies)

### A. ลดภาระการเรนเดอร์ (Rendering Optimization)
- **React.memo:** ปรับใช้กับการ์ด MVP แต่ละใบ เพื่อให้มัน Re-render เฉพาะเมื่อข้อมูลเฉพาะตัวของมันเปลี่ยนเท่านั้น
- **Virtual Lists:** หากรายการ MVP มีจำนวนมาก ให้แสดงผลเฉพาะตัวที่มองเห็นบนหน้าจอ (Viewport) เพื่อลดจำนวน DOM nodes
- **Context Splitting:** แยก `SettingsContext` ออกจาก `MvpContext` เพื่อไม่ให้การเปลี่ยนหน้าตาแอป (เช่น เปลี่ยนธีม) ไปสั่งให้ระบบคำนวณเวลาใหม่ทั้งหมด

### B. จัดการระบบนาฬิกา (Timer Management)
- **Global Clock Pattern (High Priority):** 
    - **ปัจจุบัน:** แต่ละการ์ด MVP มีนาฬิกา (Interval) ของตัวเอง ถ้ามี 50 ตัว ก็มี 50 กระบวนการ
    - **อนาคต:** สร้าง "นาฬิกากลาง" เพียงตัวเดียวในระดับแอป แล้วส่งค่าเวลาให้ทุกการ์ด "ฟัง" พร้อมกัน จะลดภาระ CPU ได้มหาศาล

### C. การจัดการทรัพยากร (Resource Efficiency)
- **Image Optimization:** บีบอัดรูปภาพ MVP และ Map เป็นฟอร์แมต **WebP** และลดความละเอียดให้พอดีกับการใช้งานจริง เพื่อลดการใช้ RAM และ GPU
- **Lazy Loading:** โหลดรูปภาพ Map เฉพาะเมื่อมีการเปิด Modal ขึ้นมาเท่านั้น

---

## 4. แผนการดำเนินงาน (Action Plan)

| ระยะ (Phase) | รายละเอียดกิจกรรม | ผลลัพธ์ที่คาดหวัง |
| :--- | :--- | :--- |
| **Phase 1: Profiling** | ใช้ React DevTools และ Lighthouse ตรวจสอบสถานะปัจจุบัน | รายงานจุดคอขวดที่ชัดเจน (Identified Bottlenecks) |
| **Phase 2: UI Sharpening** | ใช้ `memo`, `useCallback`, `useMemo` ในจุดที่วิกฤต | อาการกระตุก (Jank) หายไป UI ตอบสนองเร็วขึ้น |
| **Phase 3: Centralized Timer** | ปฏิรูประบบ `useCountdown` ให้ใช้นาฬิกากลางตัวเดียว | การใช้ CPU ลดลงอย่างเห็นได้ชัด |
| **Phase 4: Asset Polish** | บีบอัดรูปภาพและทำ Lazy Loading | แอปเบาเครื่องขึ้นและโหลดหน้าแรกเร็วขึ้น |

---

## 5. จุดที่คาดว่าเป็นสาเหตุของความหน่วงในปัจจุบัน
1. **State Update Ripple:** การอัปเดตวินาทีของ MVP แต่ละตัวอาจส่งผลให้ Component แม่ (Main Page) ต้องคำนวณงานบางอย่างซ้ำๆ
2. **Heavy Images:** ไฟล์รูปภาพ Map ที่มีความละเอียดสูงอาจกำลังกิน RAM ของ WebView มากเกินไป
3. **Timer Overhead:** การรัน `setInterval` จำนวนมากพร้อมกันในหน้าเดียว
