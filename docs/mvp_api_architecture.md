# MVP Global Intelligence API: Architectural Draft

โครงสร้างพื้นฐานสำหรับการเปลี่ยนข้อมูล MVP Timer ให้กลายเป็น API บริการระดับสากล

## 1. Core Architecture
ออกแบบให้รองรับการขยายตัว (Scalable) โดยแยกส่วนข้อมูลออกจาก UI:

*   **API Layer (Backend):** พัฒนาด้วย Rust (เช่นใช้ `Axum` หรือ `Actix-web`) เพื่อความเร็วระดับ Native
*   **Data Layer (Database):** เปลี่ยนจาก JSON file มาเป็น `PostgreSQL` หรือ `SQLite` (ถ้าต้องการความเรียบง่ายและเป็นไฟล์เดียว)
*   **Synchronization Service:** ระบบที่รับข้อมูลจากแอปของ User (ที่รัน OCR) แล้วส่งเข้ามาอัปเดตยัง Database กลางแบบ Real-time

## 2. API Endpoints ที่สำคัญ
*   `GET /servers/{server_id}/mvps`: ดึงข้อมูลสถานะบอสทั้งหมดในเซิร์ฟเวอร์
*   `GET /servers/{server_id}/mvps/{boss_id}/prediction`: ดึงข้อมูลการพยากรณ์เวลาเกิดถัดไป
*   `POST /sync/mvp-kill`: Endpoint สำหรับรับข้อมูลการตายของบอสจากแอปของผู้เล่น (ต้องมีระบบ Authentication เพื่อป้องกันข้อมูลขยะ)
*   `GET /analytics/server-trends`: ดึงข้อมูลสถิติพฤติกรรมบอสในแต่ละวัน

## 3. Data Integration (The "Global" aspect)
*   **Distributed Client:** ทุกคนที่มีแอป MVP Timer คือ "Sensor" ที่ช่วยส่งข้อมูลการเกิดบอสเข้ามายัง API กลาง
*   **Validation Logic:** มีระบบ Consensus หรือกลไกคัดกรองข้อมูล เพื่อให้แน่ใจว่าเวลาที่ส่งเข้ามาเป็นข้อมูลจริง ไม่ใช่ข้อมูลเท็จ

## 4. ความท้าทายหลัก
*   **Security:** ต้องมีการตรวจสอบข้อมูลที่ส่งเข้ามา (Rate limiting, API keys) เพื่อป้องกันการโจมตีหรือการปั่นข้อมูล
*   **Scalability:** หากผู้เล่นเพิ่มขึ้น ระบบต้องรองรับการอ่านและเขียนข้อมูลจำนวนมากพร้อมกัน
*   **Consistency:** ทำอย่างไรให้ข้อมูลของบอสตัวเดียวกันจากผู้เล่นหลายคนในเซิร์ฟเวอร์เดียวกันตรงกัน
